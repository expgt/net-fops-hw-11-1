### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. 

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему? 
 
1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.

<font color="red">Ответ:
Для гарантии целостности и поддержания чёткой структуры данных лучше всего подходят реляционные базы данных (SQL), например PostgreSQL.
Они используют концепцию ACID (атомарность, согласованность, изолированность, долговечность), которая исключает потерю или искажение данных.</font>

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

Ответ:
Если операции хеширования происходят при частых запросах одних и тех же данных, то можно использовать in-memory хранилища с доступом по ключу (API для Redis или Memcached).

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.

Ответ:
Для лендингов критически важна скорость загрузки и гибкость, а в CRM-системах преобладают сложный взаимосвязанный учет,
поэтому для данной задачи, с учетом уклона требования в сторону гибкости и быстроты, оптимально использовать связку 
документно-ориентированной СУБД (NoSQL) и реляционной СУБД с поддержкой JSON, например MongoDB и PostgreSQL

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

Ответ:
Да, данную задачу можно закрыть одной реляционной СУБД с поддержкой JSON, например PostgreSQL. Поддержка данных JSON, позволяет найти компромисс в сочетании строгой отчетность и гибкости.

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

Ответ:
Для простой и понятной структуры лучше всего использовать реляционные СУБД. В них вся информация хранится в виде привычных двумерных таблиц, 
что делает логику данных прозрачной и легкой для восприятия.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?

Ответ:
Да, можно использовать уже существующую PostgreSQL, достаточно создать одну или несколько простых таблиц.

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.

Ответ:
Для быстрой работы со связями лучше всего подходят графовые СУБД, например Neo4j. Они хранят данные в виде узлов и связей, обрабатывая отношения между ними мгновенно.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД 
логистов?

Ответ:
Да, закупки и логистика — это связанные процессы, поэтому лучше использовать одну общую СУБД, но разграничить к ней доступ.

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Ответ:
Да, для всех перечисленных задач можно использовать реляционную СУБД, например PostgreSQL. Но надо понимать, что при определенных сценариях данное решение может быть неоптимальным,
особенно в долгосрочной перспективе. Основные недостатки такого подхода это: 
- Проблемы с масштабируемостью
- Неэффективность при работе с разными типами данных
- Сложность поддержки
- Единая точка отказа
- Ограниченный выбор инструментов

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

Ответ:
Пополнение баланса телефона — это классическая финансовая транзакция, которая в биллинговой системе мобильного оператора или банка должна выполняться атомарно (все шаги успешны) либо откатываться целиком (если произошел сбой).
В реляционной базе данных (БД) процесс проходит через следующие шесть шагов:

- Инициализация транзакции (Начало сессии):
  Система открывает сессию БД и начинает транзакцию (например, команда BEGIN), устанавливая уровень изоляции, чтобы избежать конфликтов с другими операциями (например, READ COMMITTED или SERIALIZABLE).
  
- Проверка существования абонента:
  Выполняется SELECT запрос для проверки номера телефона в таблице абонентов. Система верифицирует, что номер существует, активен и готов принимать платежи. Если номер не найден, транзакция прерывается.
  
- Блокировка записи (Row-level Locking):
  Накладывается эксклюзивная блокировка на запись баланса конкретного абонента. Это критически важно, чтобы предотвратить "состояние гонки" (race condition), если абонент попытается пополнить счет в двух местах одновременно.

- Списание средств (для оплат с кошелька/карты) или генерация записи транзакции:
  Совершается UPDATE баланса или INSERT записи о платеже (в зависимости от архитектуры — предоплата или отложенный платеж). В этой же операции уменьшается баланс кошелька/банковского счета плательщика, если оплата идет через внутренний сервис.

- Фиксация транзакции (Commit):
  После успешного изменения данных система выполняет команду COMMIT. Все изменения записываются на диск (в журнал транзакций WAL), данные становятся постоянными, а блокировка с записи снимается.

- Уведомление и логирование (Финализация):
  Флаг статуса транзакции меняется на «Успешно» (SUCCESS), записывается финальный лог для аудита. После этого система отправляет push-уведомление или SMS абоненту о том, что баланс успешно пополнен.
  

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

Ответ:
Если пополнение происходит через автоплатёж, добавляются фоновые процессы:

- Триггер события: 
  Автоматизированная система мониторинга (планировщик) ежедневно проверяет биллинг-базу. Если баланс абонента опускается ниже установленного порога (например, 30 рублей), инициируется процесс автоплатежа.

- Запрос токена (Связка с банком): 
  Система биллинга не хранит данные карты, она отправляет запрос в процессинговый центр с использованием ранее сохраненного токена карты (привязанного в приложении) для списания средств.

- Автоматическая транзакция: 
  В БД создается запись об инициировании пополнения, а после успешного ответа от банка выполняются стандартные 6 шагов, описанные выше.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.

Ответ:
Преимущества SQL-систем по отношению к NoSQL.

- Строгая согласованность данных (ACID):
  Гарантируют абсолютную целостность. Любая транзакция либо полностью завершается, либо отменяется, что исключает появление «осиротевших» или противоречивых записей, критически важных для финансов и бухгалтерии.

- Сложные и мощные соединения (JOIN):
  Позволяют эффективно извлекать и объединять данные из множества таблиц за один запрос, используя всю мощь математической реляционной алгебры.

- Стандартизированный язык (SQL):
  Благодаря единому стандарту вы можете легко переносить свои знания или менять саму СУБД (например, с PostgreSQL на MySQL) без необходимости переписывать всю логику взаимодействия с базой.

- Надежность и атомарность (целостность схемы):
  Жесткая структура требует соблюдения определенных типов данных и связей (внешние ключи). Это защищает базу от случайного ввода некорректной или «мусорной» информации на уровне архитектуры.

- Универсальность для сложной аналитики:
  Идеально подходят для комплексных отчетов, агрегаций и группировок благодаря встроенным математическим и аналитическим функциям, которые в NoSQL требовали бы сложного программирования на стороне приложения.
  

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

Ответ:
Преимущества перед SQL-системами

- Неограниченная горизонтальная масштабируемость:
  Базы данных NewSQL автоматически распределяют (шардируют) данные по разным узлам кластера по мере роста нагрузки, избавляя от необходимости ручного масштабирования.
- Отказоустойчивость:
  Выход из строя одного или нескольких серверов не приводит к простою системы или потере данных благодаря автоматической репликации.
- Высокая доступность:
  Архитектура создавалась с учетом облачных вычислений, поэтому кластер работает непрерывно («из коробки»), что снимает нагрузку с администраторов.

Преимущества перед NoSQL-системами

- Строгая согласованность и транзакции (ACID):
  В отличие от NoSQL, где данные часто обновляются с задержкой (eventual consistency), NewSQL гарантирует нерушимость транзакций, что критически важно для финансовых, банковских и бухгалтерских систем.
- Привычный язык SQL и сложные соединения (JOIN):
  NoSQL часто ограничивает разработчиков базовыми операциями "ключ-значение". NewSQL сохраняет мощь SQL-запросов, позволяя легко работать со сложными выборками, связями и аналитикой без необходимости реализовывать логику на уровне приложения.
- Поддержка схемы данных: 
  Дает возможность жестко структурировать данные и использовать вторичные индексы без ущерба для скорости.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

Ответ:
Для обработки большого количества информации на 1000 машинах классические СУБД (например, PostgreSQL) неэффективны. 
Выбор будет определяться характером нагрузки и требованиями к консистентности данных.
Основные критерии выбора: тип обработки (OLAP/OLTP), скорость отклика (real-time vs batch) и структура данных.

Оптимальный выбор модели распределённых вычислений для данной задачи — вычисления в оперативной памяти (In-Memory Computing).
Преимущества:
- Скорость (In-Memory): данная модель кэширует промежуточные результаты вычислений в оперативной памяти (RAM) узлов,
  выполняет итерактивные алгоритмы, машинное обучение и графовые задачи в десятки и сотни раз быстрее, чем устаревшие дисковые модели.
- Устойчивость к отказам (RDD): Использование отказоустойчивых распределенных датасетов (RDD) гарантирует, что при выходе из строя части машин данные не потеряются, а вычисления автоматически пересчитаются.
